# KP B1.5 — Pipes · Exit Status · Command Grouping
**`|` · `$?` · `&&` `||` `;` · pipeline exit / `pipefail` / `PIPESTATUS` · `{ }` vs `( )`**

**10 tasks · ⭐⭐ · RHEL 10 · server1 as root · run each in a bash shell and match the stated result**

> Whether a script fails silently comes down to whether you understand **exit codes**. 0 = success, non-zero = failure; `&&`/`||` branch on exactly that. Make sure tasks 6–7 (the **pipeline exit-code trap**) and tasks 8–9 (`()` vs `{}`) really land.

---

**1.** With a two-stage pipe, count how many accounts in `/etc/passwd` use `/bin/bash` (`grep … | wc -l`).
*Acceptance:* prints a number (varies by host); the command is a pipeline `grep '/bin/bash$' /etc/passwd | wc -l`.

**2.** Show `$?`: run a command that succeeds, print `$?`; run one that fails, print `$?`.
*Acceptance:* `0` after the success, a non-zero value (e.g. `2`) after the failure.

**3.** `&&` (run-if-success): write `mkdir -p /tmp/x && echo created` so the `echo` runs **only because** `mkdir` succeeded.
*Acceptance:* prints `created`.

**4.** `||` (run-if-failure): write `ls /nope 2>/dev/null || echo "not found"` so the `echo` runs **only because** `ls` failed.
*Acceptance:* prints `not found`.

**5.** `;` vs `&&`: show that `false; echo A` prints `A` (sequence runs unconditionally) but `false && echo A` prints nothing.
*Acceptance:* first prints `A`; second prints nothing.

**6.** Pipeline exit code (default) = the **last** stage. Run `false | true`, then in **one** command print `$?` and `PIPESTATUS`.
*Acceptance:* `$?` is `0` (the failure of `false` is hidden); `PIPESTATUS` is `1 0`.

**7.** `pipefail`: enable `set -o pipefail`, then run `false | true` and print `$?`.
*Acceptance:* now `$?` is `1` (the pipeline reports the failing stage); `PIPESTATUS` is still `1 0`.

**8.** Subshell isolation `( )`: run `x=1; ( x=99; cd /tmp ); echo "$x $PWD"`.
*Acceptance:* after the `( )`, `x` is still `1` and `$PWD` is unchanged — changes inside a subshell don't leak out.

**9.** Grouping in the **current** shell `{ }`: send the combined output of two `echo`s to a file with `{ echo a; echo b; } > both.txt`. (Mind the spaces after `{` and the `;` before `}`.)
*Acceptance:* `both.txt` has 2 lines; and you can say why `{ }` (same shell) differs from `( )` (subshell).

**10.** The `A && B || C` pitfall: show `cmd && echo yes || echo no` prints `yes` when `cmd` succeeds and `no` when it fails — **but** explain why this is *not* a safe `if/else`.
*Acceptance:* success → `yes`, failure → `no`; your answer explains the trap (if `B` itself fails, `C` also runs) and when to use a real `if` instead.
